## 二、架构分析

### 2.1 优点

- **分层架构清晰**: `core → data_layer → strategy_engine → backtest_engine → optimizer → signal_output → ui`, 依赖方向单一, 无循环依赖
- **策略工厂模式**: `StrategyFactory` 注册+动态创建, 扩展新策略只需注册, 符合开闭原则(OCP)
- **数据层DB优先策略**: `DataManager` 优先使用本地缓存, 减少网络请求, 增量更新机制合理
- **多数据源容错**: `DataFetcher` 三源降级(腾讯→新浪→东方财富), 提高数据获取可靠性
- **A股规则完整**: T+1、佣金+印花税、涨跌停、最小100股, 回测贴近真实交易

### 2.2 问题

#### P1: BacktestRunner 重复构建 BacktestResult (DRY违反)

`run_backtest()` 和 `run_fusion_backtest()` 中构建 `BacktestResult` 的代码几乎完全相同(约30行), 仅 `strategy_type` 和 `params` 不同。

**建议**: 提取 `_build_result()` 方法:

```python
def _build_result(self, symbol, strategy_type, params, start_date, end_date,
                  trades, equity_curve, metrics) -> BacktestResult:
    return BacktestResult(
        symbol=symbol, strategy_type=strategy_type, params=params,
        start_date=start_date, end_date=end_date,
        trades=trades,
        total_trades=len(trades),
        win_trades=sum(1 for t in trades if t.profit > 0),
        loss_trades=sum(1 for t in trades if t.profit <= 0),
        **metrics,
        equity_curve=equity_curve,
    )
```

#### P2: SignalGenerator 两个方法逻辑完全重复

`generate_signals()` 和 `generate_historical_signals()` 代码完全相同, 仅参数名不同。

**建议**: 合并为一个方法, 或让 `generate_signals` 调用 `generate_historical_signals`:

```python
def generate_signals(self, symbol, end_date):
    return self.generate_historical_signals(symbol, "2020-01-01", end_date)
```

#### P3: 4个策略类结构高度雷同

每个策略都有相同的 `__init__` 模式、`IndicatorCalculator` 实例化、`calc_all_indicators` 调用、逐行遍历生成信号。可提取公共模板方法到 `BaseStrategy`。

**建议**: 在 `BaseStrategy` 中增加模板方法:

```python
def generate_signals(self, df):
    df_ind = self._calc.calc_all_indicators(df, self.params)
    return self._evaluate_signals(df_ind)

@abstractmethod
def _evaluate_signals(self, df_ind) -> List[Signal]:
    """子类只需实现信号评估逻辑"""
```

#### P4: MainWindow 职责过重 (SRP违反)

`MainWindow` 承担了所有业务协调: 数据获取、策略配置、回测、优化、信号生成、UI交互。490行代码, 15+个事件处理方法。

**建议**: 引入 `AppController` 或 `ServiceLocator`, 将业务逻辑从UI层剥离:

```python
class AppController:
    """业务协调器, MainWindow只负责UI"""
    def run_backtest(self, symbol, config): ...
    def run_optimization(self, symbol, config): ...
    def generate_signals(self, symbol): ...
```

## 四、性能分析

### 4.1 K线图重绘性能 (严重)

`KlineWidget._update_chart()` 每次操作(缩放、滚动、指标切换)都:
1. 销毁旧canvas (`deleteLater`)
2. 重新调用 `mpf.plot()` 生成全新Figure
3. 创建新 `FigureCanvas`
4. 添加到布局

这是**极其低效**的做法。每次交互都完整重绘, 包括指标计算、信号匹配、matplotlib渲染全流程。

**建议**: 使用 `blitting` 技术或仅更新数据而非重建Figure:

```python
# 方案1: 保留Figure, 只更新axes数据
def _update_chart(self):
    if self._canvas is None:
        # 首次创建
        fig, axes = mpf.plot(...)
        self._canvas = FigureCanvas(fig)
    else:
        # 后续只更新数据范围
        self._axes[0].set_xlim(start, end)
        self._canvas.draw_idle()
```

### 4.2 策略逐行遍历 (中等)

4个策略都使用 `for i in range(...)` 逐行遍历DataFrame, 这是pandas反模式。

**建议**: 使用向量化操作替代逐行遍历:

```python
# 技术指标策略 - 向量化MACD金叉检测
macd_cross_up = (dif.shift(1) <= dea.shift(1)) & (dif > dea) & (hist > 0)
buy_mask = macd_cross_up | (rsi_6 < rsi_buy) | (close <= boll_lower)
buy_score = buy_mask.astype(int).groupby(level=0).sum()
```

### 4.3 数据库单连接 (中等)

`DatabaseManager` 使用单个 `sqlite3.Connection`, 且 `check_same_thread=False`, 在多线程环境下(QThread workers)存在竞态条件。

**建议**: 使用连接池或每线程独立连接:

```python
import threading

class DatabaseManager:
    def connect(self):
        # 每线程独立连接
        thread_id = threading.get_ident()
        if thread_id not in self._connections:
            conn = sqlite3.connect(str(self._db_path))
            conn.row_factory = sqlite3.Row
            conn.execute("PRAGMA journal_mode=WAL")
            self._connections[thread_id] = conn
        return self._connections[thread_id]
```

### 4.4 calc_all_indicators 每次全量计算

`DataManager.get_data_with_indicators()` 每次调用都重新计算所有指标, 即使数据未变。

**建议**: 缓存带指标的数据, 或将指标列持久化到数据库:

```python
def get_data_with_indicators(self, symbol, start_date, end_date):
    df = self.ensure_data_available(symbol, start_date, end_date)
    # 检查是否已有指标列
    if "macd_dif" in df.columns:
        return df
    return self._indicator_calc.calc_all_indicators(df)
```

## 五、安全性分析

### 5.1 SQL注入风险

| 位置 | 问题 |
|------|------|
| `database.py:211` | `INSERT OR REPLACE INTO {table}` — table名来自参数, 无校验 |
| `database.py:220-227` | `UPDATE {table} SET {set_clause}` — 动态拼接列名 |
| `database.py:246` | `SELECT COUNT(*) as cnt FROM {table_name}` — f-string拼接 |

虽然当前调用方传入的都是固定表名, 但API设计上存在注入风险。

**建议**: 添加表名白名单校验:

```python
_VALID_TABLES = {"daily_kline", "stock_info", "signal_records", ...}

def insert_batch(self, table, data):
    if table not in _VALID_TABLES:
        raise ValueError(f"Invalid table: {table}")
    ...
```

### 5.2 线程安全

- `DatabaseManager` 单连接 + `check_same_thread=False` = 竞态条件
- `StrategyFactory._registry` 是类变量, 多线程注册/创建不安全
- Worker线程的 `_stop_flag` 无锁保护

**建议**: Worker的stop机制使用 `threading.Event`:

```python
class BacktestWorker(QThread):
    def __init__(self, ...):
        self._stop_event = threading.Event()

    def run(self):
        while not self._stop_event.is_set():
            ...

    def stop(self):
        self._stop_event.set()
```

### 5.3 输入校验缺失

- 股票代码输入无格式校验(应为6位数字)
- 日期范围无逻辑校验(结束日期应大于开始日期)
- 策略参数范围校验(`validate_params`)未被调用

## 六、功能缺陷分析

### 6.1 均值回归策略阈值逻辑错误

`mean_reversion_strategy.py:72`:

```python
if deviation < -deviation_threshold / 100 and prev_deviation < deviation:
```

`deviation_threshold` 默认值2.0, 除以100后变成0.02(2%), 但 `deviation` 本身已经是百分比形式(如-0.03表示-3%)。这意味着阈值实际是2%, 而配置意图可能是2个标准差。逻辑与BOLL策略的偏离度概念不一致。

**建议**: 明确偏离度的定义(百分比 vs 标准差倍数), 统一阈值含义:

```python
# 如果deviation是百分比形式, 阈值应直接用百分比值
if deviation < -deviation_threshold and prev_deviation < deviation:
    # deviation_threshold = 0.02 表示偏离2%
```

### 6.2 融合引擎未处理无信号日期

`strategy_fusion.py` 只处理有信号的日期, 不产生HOLD信号。这意味着融合结果中缺少大量"观望"日期, 导致:
- 回测时只关注有信号的日期, 忽略了持仓期间的无信号日
- K线图上只标注BUY/SELL, 无法看到HOLD状态

**建议**: 对每个交易日都产生融合信号(包括HOLD):

```python
for date in df["date"]:
    sigs = date_signals.get(date, [])
    # 即使无策略信号, 也产生HOLD
    ...
```

### 6.3 权益曲线计算与交易模拟不一致

`trade_simulator.py` 的 `get_equity_curve()` 简化处理:
- 未考虑佣金对权益的影响
- `exit_date` 匹配包含"(未平仓)"标记, 导致匹配失败
- 买入股数重新计算而非使用trade记录中的实际值

**建议**: 权益曲线应直接从交易模拟过程中生成, 而非事后重建:

```python
def simulate_trades(self, signals, df):
    trades = []
    equity_curve = []
    capital = self.initial_capital
    position = 0

    for _, row in df.iterrows():
        # 处理信号...
        # 每日记录权益
        equity_curve.append((row["date"], capital + position * row["close"]))

    return trades, equity_curve
```

### 6.4 遗传算法未记录中间结果

`genetic_optimizer.py` 的 `all_results=[]` 为空列表, 无法查看进化过程和中间参数。

**建议**: 在每代进化后记录统计信息:

```python
all_results = []
for gen in range(generations):
    ...
    all_results.append({
        "generation": gen,
        "avg_fitness": stats["avg"],
        "max_fitness": stats["max"],
        "best_params": param_space.decode_vector(hof[0]),
    })
```

### 6.5 数据归一化单位混淆

`data_fetcher.py:366-368`:

```python
if not has_volume and has_amount:
    df["volume"] = df["amount"] * 100  # 假设amount是手数
    df["amount"] = df["volume"] * df["close"]
```

腾讯源的 `amount` 列实际含义是"成交额(元)"而非"成交量(手)", 这个推导逻辑会导致成交量被放大100倍。

**建议**: 根据数据源区分处理, 或在归一化时明确标注单位:

```python
# 腾讯源: volume列不存在, amount=成交额(元)
if source == "腾讯":
    df["volume"] = df["amount"] / df["close"]  # 反推成交量
```

## 九、总结

### 核心优势
1. **功能闭环完整**: 从数据获取到信号输出, 全流程可运行, 已有真实使用数据
2. **A股规则贴合**: T+1、佣金印花税、涨跌停处理, 回测贴近真实
3. **多策略融合**: 4策略加权投票, 可配置权重和阈值
4. **多数据源容错**: 三源降级+重试机制, 数据获取可靠性高
5. **UI交互完善**: 深色/浅色主题、进度对话框、键盘导航、交易标注

### 核心风险
1. **策略逻辑准确性**: 均值回归阈值错误、数据单位混淆, 直接影响信号质量
2. **性能瓶颈**: K线图每次交互都完整重建, 大数据量下体验差
3. **线程安全**: 数据库单连接+多线程, 存在竞态条件
4. **测试空白**: 无正式测试, 代码变更无质量保障

### 建议路线图
1. **第一阶段(1周)**: 修复P0逻辑错误 + P1性能和安全问题
2. **第二阶段(2周)**: P2代码质量改善 + 融合引擎改进
3. **第三阶段(持续)**: 建立测试体系 + 文档补充 + 架构优化

项目功能完整度较高, 但工程化成熟度不足。优先修复影响信号准确性的逻辑错误, 再系统性提升代码质量和性能。
